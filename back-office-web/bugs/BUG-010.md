# 🐛 Bugfix Spec: Default de canales no se aplica en flujo collector al cargar políticas

> **ID:** BUG-010
> **Fecha:** 2026-02-26
> **Proyecto:** back-office-web
> **Prioridad:** critical
> **Estado:** resolved

---

## 1. Resumen

El fix de default de canales (`appliesPortal: true, appliesMicrosite: true`) al cargar políticas desde la API solo se aplicaba en el flujo no-collector (`!isCollector`), dejando el flujo collector sin la corrección y potencialmente causando que el botón "Guardar cambios" quede deshabilitado.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/adminSettings/TermsAndConditions/hooks/useCreatePolicie.tsx` | Carga y mapea `policies_data` al estado inicial del formulario (Formik) |
| `src/presentation/adminSettings/TermsAndConditions/components/PoliciesPage/CreatePolicie/CreatePolicie.tsx` | Renderiza el botón "Guardar cambios" con `disabled={isSubmitting \|\| !dirty}` |

### Dependencias clave

- Componente padre: `PoliciesPage`
- Store/slice: `policies` (Redux)
- Hook: `useCreatePolicie(isCollector: boolean)`
- Formik: `enableReinitialize={true}` — reinicializa si `initialFormValues` cambia

---

## 3. Reproducción

### Pasos

1. Ir a `/configuracion/terminos-y-condiciones` en el flujo **collector**
2. Cargar la página con políticas existentes que tengan `appliesPortal: false` y `appliesMicrosite: false`
3. Observar el estado del botón "Guardar cambios"

### Comportamiento actual

El default de canales (`appliesPortal: true, appliesMicrosite: true`) no se aplica en el flujo collector, dejando los valores tal como vienen de la API. Esto puede provocar que Formik no detecte cambios (`dirty: false`) y el botón "Guardar cambios" permanezca deshabilitado.

### Comportamiento esperado

Tanto en flujo collector como no-collector, si una política cargada desde la API tiene `appliesPortal: false` y `appliesMicrosite: false`, se deben establecer ambos en `true` como valor por defecto del formulario.

---

## 4. Análisis de causa raíz

### Hipótesis

- [x] El ternario `!isCollector ? { fix } : mapped` excluye el flujo collector del fix de defaults de canales.

### Causa confirmada

En `useCreatePolicie.tsx`, el mapeo de `policies_data` al estado inicial aplica el default de canales condicionalmente:

```ts
// ❌ Solo aplica el fix para !isCollector
const formValues = !isCollector
  ? {
      policies: mapped.policies.map((p) =>
        !p.appliesPortal && !p.appliesMicrosite
          ? { ...p, appliesPortal: true, appliesMicrosite: true }
          : p,
      ),
    }
  : mapped; // ← collector usa mapped sin el fix
```

Para collector, `mapped` se usa directamente sin normalizar los canales, reproduciendo el bug de defaults incorrectos.

**Archivo raíz:** `src/presentation/adminSettings/TermsAndConditions/hooks/useCreatePolicie.tsx`
**Líneas:** `47-55`

---

## 5. Plan de fix

### Estrategia

Eliminar el ternario `!isCollector` y aplicar el fix de defaults de canales de forma incondicional para ambos flujos.

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `src/presentation/adminSettings/TermsAndConditions/hooks/useCreatePolicie.tsx` | Reemplazar ternario `!isCollector` por mapeo incondicional con defaults de canales | SRP |

### Restricciones

- ❌ No modificar `CreatePolicie.tsx` ni la condición `disabled` del botón
- ❌ No alterar el bloque `urlToBase64` que sigue al `setInitialFormValues`
- ✅ El fix aplica igual para collector y no-collector

# 🐛 Bugfix Spec: Guardar política falla cuando solo tiene URL (sin PDF)

> **ID:** BUG-008
> **Fecha:** 2026-02-26
> **Proyecto:** back-office-web
> **Prioridad:** high
> **Estado:** resolved

---

## 1. Resumen

Al crear una política con adjunto tipo URL (sin PDF), al hacer clic en "Guardar cambios" y confirmar, la política no se guarda. El sistema permite solo una opción de adjunto: URL o PDF, y ambas deben funcionar de forma independiente.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/adminSettings/TermsAndConditions/helpers/formValidator.ts` | Valida `linkUrl` con `urlRegex` (requiere `https://`) |
| `src/presentation/adminSettings/TermsAndConditions/components/AttachmenComponent/constants/constants.ts` | Define `urlPattern` (NO requiere `https://`) usado en el botón "Aplicar" |
| `src/presentation/adminSettings/TermsAndConditions/helpers/createPolicyAdapter.ts` | `buildLink` construye el payload; para `linkType === "url"` usa `p.linkUrl` |

| `src/presentation/adminSettings/TermsAndConditions/components/AttachmenComponent/hooks/usePolicyAttachmentCollapse.ts` | `validateButton` usa `urlPattern`; `handleApplyClick` escribe `linkUrl` en Formik |

### Dependencias clave

- Store/slice: `policies` (Redux)
- Hooks: `useCreatePolicie`, `usePolicyAttachmentCollapse`, `usePolicyItem`

## 3. Reproducción

### Pasos

1. Ir a `/configuracion/terminos-y-condiciones`
2. Hacer clic en "Agregar política"
3. Completar título y contenido
4. Seleccionar el radio button **URL/Link**
5. Ingresar una URL **sin protocolo** (ej: `www.ejemplo.com` o `ejemplo.com`)
6. Completar el campo de nombre del link
7. Hacer clic en "Aplicar" → el botón se habilita y cierra el card
8. Hacer clic en "Guardar cambios" → modal abre
9. Hacer clic en "Confirmar"

### Comportamiento actual

La política no se guarda. El modal se cierra y el formulario vuelve a mostrar errores de validación en el campo URL, pero sin mensaje visible claro para el usuario de que el problema fue el formato de URL.

### Comportamiento esperado

- Si la URL ingresada es válida (con o sin protocolo), la política debe guardarse.
- Si la URL no es válida, el error debe mostrarse ANTES de llegar al modal de confirmación (o con el mismo criterio que el botón "Aplicar").

## 4. Análisis de causa raíz

### Hipótesis

- [x] **H1 — Discrepancia entre `urlPattern` y `urlRegex`:** El botón "Aplicar" usa `urlPattern` (que acepta URLs sin protocolo como `www.ejemplo.com`). Formik valida con `urlRegex` que **exige** `https://` o `http://`. El usuario puede pasar "Aplicar" con `www.ejemplo.com`, pero Formik marca `isValid = false`. `handleClickConfirm` ve `isValid = false`, cierra el modal y activa `setTouched` mostrando errores en el form, **sin guardar nada**.

- [ ] **H2 — `linkUrl` vacío en Formik al momento del submit:** Si el usuario no hace clic en "Aplicar" y va directo a "Guardar", `handleLinkChange` ya actualizó `linkUrl` en Formik vía `onLinkChange` en cada keystroke. Esta hipótesis es menos probable pero merece verificación.

- [ ] **H3 — `onFilesChange([], 0)` en `handleApplyClick` causa race condition con `setFieldValue`:** Tres llamadas seguidas a `setFieldValue` (`files`, `fileSize`, `originalS3Url`) podrían causar que un render intermedio sobreescriba el `linkUrl` recién escrito. Requiere testing específico.

### Causa confirmada

**H1 es la causa principal confirmada por análisis estático.**

```
urlPattern = /^(?:https?:\/\/)?(?:www\.)?[a-z\d-]+.../i
  → Acepta: www.ejemplo.com, ejemplo.com, https://ejemplo.com

urlRegex = /^(https?:\/\/)([\w-]+(\.[\w-]+)+).../i
  → Requiere: https:// o http://
  → Rechaza: www.ejemplo.com, ejemplo.com
```

El usuario ingresa `www.ejemplo.com`:
- `validateButton()` → `urlPattern.test("www.ejemplo.com")` = `true` → botón Aplicar HABILITADO
- Formik `validateUrl` → `urlRegex.test("www.ejemplo.com")` = `false` → `errors.linkUrl = "urlInvalid"` → `isValid = false`
- `handleClickConfirm(isValid=false, ...)` → `onApplyCreate()` + `setTouched()`, return → **no guarda**

**Archivos raíz:**
- `src/.../helpers/formValidator.ts` — línea 5 (`urlRegex`) y líneas 68-72 (`validateUrl`)
- `src/.../AttachmenComponent/constants/constants.ts` — línea 20-21 (`urlPattern`)

## 5. Plan de fix

### Estrategia

Unificar los criterios de validación de URL entre el botón "Aplicar" y Formik. Hay dos opciones:

**Opción A (recomendada):** Hacer que `urlRegex` en `formValidator.ts` acepte URLs sin protocolo, igual que `urlPattern`. Así el usuario puede ingresar `www.ejemplo.com` y guardar sin problemas.

**Opción B:** Hacer que el botón "Aplicar" use `urlRegex` (más estricto) para que el error se muestre a nivel del componente antes de llegar a "Guardar".

La Opción A es más permisiva y mejor UX (el backend puede normalizar la URL). La Opción B es más estricta pero podría ser confusa si el backend acepta URLs sin protocolo.

### Cambios requeridos (Opción A)

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `src/.../helpers/formValidator.ts` | Reemplazar `urlRegex` por `urlPattern` importado desde constants, o unificar regex para que no exija protocolo | SRP |

### Restricciones

- ❌ No modificar `urlPattern` en `constants.ts` (usado por el botón Aplicar; si se hace más estricto se necesita verificar UX)
- ❌ No tocar el flujo S3 (PDF)
- ❌ No modificar `PolicyLinkRequest` ni el adapter `buildLink` (el caso URL ya construye el payload correctamente)
- ✅ Solo cambiar el regex de validación en `formValidator.ts`

## 8. Resolución

> ⚠️ A completar DESPUÉS del fix.

### Fecha de resolución

2026-02-26

### Cambios realizados

```diff
// formValidator.ts
- const urlRegex = /^(https?:\/\/)([\w-]+(\.[\w-]+)+)(:\d{2,5})?([/?#][^\s]*)?$/i;
+ const urlRegex = /^(?:https?:\/\/)?(?:www\.)?[a-z\d-]+(?:\.[a-z\d-]+)*\.[a-z]{2,}(?::\d+)?(?:[/?#][^\s]*)?$/i;
```

### Lecciones aprendidas

- **Patrón del bug:** Doble estándar de validación — mismo campo validado con criterios distintos en UI vs Formik
- **Skill relacionada:** `components/formValidator`
- **Regla nueva:** Cualquier validación de formato en un campo de entrada debe usar el mismo regex/criterio que la validación de Formik. Nunca tener dos standards distintos para el mismo campo.

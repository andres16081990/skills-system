# 🐛 Bugfix Spec: PolicyItem no valida caracteres permitidos en modo Collector

> **ID:** BUG-002
> **Fecha:** 2026-02-25
> **Proyecto:** back-office-web
> **Prioridad:** medium
> **Estado:** resolved

---

## 1. Resumen

En `PolicyItem`, cuando `isCollector=true`, el campo `OutlinedInput` (título) y el `RichTextEditor` (contenido adicional) no restringen los caracteres permitidos ni en la UI ni en la validación Formik.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/adminSettings/TermsAndConditions/components/PolicyItem/PolicyItem.tsx` | UI: renderiza `OutlinedInput` y `RichTextEditor` |
| `src/presentation/adminSettings/TermsAndConditions/hooks/usePolicyItem.tsx` | Hook Formik del item — no recibe ni propaga `isCollector` |
| `src/presentation/adminSettings/TermsAndConditions/helpers/formValidator.ts` | Validaciones Formik — `validateTitle` y `validateContent` no tienen reglas de caracteres |

### Dependencias clave

- Componente padre: `CreatePolicie.tsx` (pasa `isCollector`)
- Validador: `validateSinglePolicy(p, isCollector, translate)` → llama a `validateTitle` y `validateContent` que **no reciben `isCollector`**
- Hook: `usePolicyItem(index, policy, remove, setSubmitting)` — no recibe `isCollector`

---

## 3. Reproducción

### Pasos

1. Ir a `/configuracion/terminos-y-condiciones/collector`
2. Crear una nueva política
3. En el campo de título (`OutlinedInput`) ingresar caracteres como `!`, `@`, `#`, `$`, `%`, números, etc.
4. En el `RichTextEditor` ingresar los mismos caracteres especiales no permitidos
5. Intentar guardar

### Comportamiento actual

- El `OutlinedInput` acepta cualquier carácter (solo limita a 100 con `maxLength`)
- El `RichTextEditor` acepta cualquier carácter (solo limita a 2000 con `maxLength`)
- `validateTitle` únicamente verifica si el campo está vacío
- `validateContent` únicamente verifica si `contentLength` es 0 o supera 2000

### Comportamiento esperado

**`OutlinedInput` (título) — modo Collector:**
- Máximo 100 caracteres ✅ (ya está)
- Solo permite: letras (a-z, A-Z, con tildes/diacríticos), espacios, `&`, `-`, `_`
- Regex sugerida: `/^[a-zA-ZÀ-ÿ\s&\-_]+$/`
- Error de validación si contiene caracteres fuera de ese set

**`RichTextEditor` (contenido adicional) — modo Collector:**
- Máximo 2000 caracteres ✅ (ya está via `MAX_LENGTH_TEXT`)
- Solo permite: letras (con tildes/diacríticos), espacios, `&`, `-`
- Regex sobre el texto plano (sin tags HTML): `/^[a-zA-ZÀ-ÿ\s&\-]+$/`
- Error de validación si contiene caracteres fuera de ese set

---

## 4. Análisis de causa raíz

### Hipótesis

- [x] **H1 — `validateTitle` no tiene validación de caracteres:** La función no recibe `isCollector` y solo hace `isBlank()`. Falta agregar regex check condicional.
- [x] **H2 — `validateContent` no tiene validación de caracteres:** Igual que H1. Valida longitud pero no el set de caracteres. Para RichText hay que extraer el texto plano del HTML antes de aplicar el regex.
- [ ] **H3 — `usePolicyItem` no pasa `isCollector` al validador:** El hook no recibe `isCollector`, pero la validación ocurre en `formValidator.ts` que sí recibe el flag desde `validateSinglePolicy`.

### Causa confirmada

`validateTitle` y `validateContent` en `formValidator.ts` no aplican validación de caracteres. `validateSinglePolicy` ya recibe `isCollector` pero no lo propaga a esas dos funciones.

**Archivo raíz:** `src/presentation/adminSettings/TermsAndConditions/helpers/formValidator.ts`
**Líneas clave:**
- `validateTitle`: líneas 11–19 (sin regex de caracteres)
- `validateContent`: líneas 21–31 (sin regex de caracteres)

---

## 5. Plan de fix

### Estrategia

Agregar validación de caracteres en `formValidator.ts` para `validateTitle` y `validateContent` cuando `isCollector=true`, pasando el flag desde `validateSinglePolicy`. No modificar la UI (`PolicyItem.tsx`) para la validación — el `maxLength` en el `OutlinedInput` ya está correcto.

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `formValidator.ts` | Agregar parámetro `isCollector` a `validateTitle` y aplicar regex `/^[a-zA-ZÀ-ÿ\s&\-_]+$/` sobre `p.title` cuando sea true | SRP |
| 2 | `formValidator.ts` | Agregar parámetro `isCollector` a `validateContent` y aplicar regex `/^[a-zA-ZÀ-ÿ\s&\-]+$/` sobre el texto plano del HTML cuando sea true | SRP |
| 3 | `formValidator.ts` | En `validateSinglePolicy`, pasar `isCollector` a `validateTitle` y `validateContent` | — |

> **Nota para extraer texto plano del RichText:** usar una utilidad tipo `const plainText = html.replace(/<[^>]*>/g, "")` antes de aplicar el regex al contenido.

### Restricciones

- ❌ No modificar `PolicyItem.tsx` para la lógica de validación (la validación vive en el validator)
- ❌ No modificar la firma pública de `validateSinglePolicy` ni `validatePolicies`
- ❌ No alterar `usePolicyItem` — `isCollector` no necesita llegar al hook para este fix
- ✅ Se puede agregar constantes de regex en el mismo `formValidator.ts`
- ✅ Se pueden agregar nuevas claves i18n para los mensajes de error de caracteres

## 6. Criterios de aceptación

- [ ] En modo Collector, el título con tildes (`á`, `é`, `ñ`), `&`, `-`, `_` NO muestra error

## 8. Resolución

### Fecha de resolución

2026-02-25

### Cambios realizados

```diff
// src/presentation/adminSettings/TermsAndConditions/helpers/formValidator.ts

+ const collectorTitleRegex = /^[a-zA-ZÀ-ÿ\s&\-_]+$/;
+ const collectorContentRegex = /^[a-zA-ZÀ-ÿ\s&\-]+$/;
+ const stripHtml = (html: string) => html.replace(/<[^>]*>/g, "").replace(/&nbsp;/g, " ");

// validateTitle — agrega isCollector y validación de chars
- const validateTitle = (p, translate, errors) => {
+ const validateTitle = (p, isCollector, translate, errors) => {
    if (isBlank(p.title)) {
      errors.title = translate("validations.titleRequired");
+   } else if (isCollector && !collectorTitleRegex.test(p.title.trim())) {
+     errors.title = translate("validations.titleInvalidChars");
    }
  };

// validateContent — agrega isCollector y validación de chars sobre texto plano
- const validateContent = (p, translate, errors) => {
+ const validateContent = (p, isCollector, translate, errors) => {
    if (!p.contentLength) { ... }
    else if (p.contentLength > 2000) { ... }
+   else if (isCollector && !collectorContentRegex.test(stripHtml(p.content ?? ""))) {
+     errors.content = translate("validations.contentInvalidChars");
+   }
  };

// validateSinglePolicy — propaga isCollector
- validateTitle(p, translate, errors);
- validateContent(p, translate, errors);
+ validateTitle(p, isCollector, translate, errors);
+ validateContent(p, isCollector, translate, errors);
```

### Lecciones aprendidas

- **Patrón del bug:** Funciones de validación internas que no recibían el contexto (`isCollector`) para aplicar reglas condicionales.
- **Regla nueva:** Cuando `validateSinglePolicy` recibe un flag de contexto, verificar que lo propaga a **todas** las sub-funciones de validación que lo necesiten.
- **Tip para RichText:** Antes de aplicar regex de caracteres sobre contenido HTML, usar `stripHtml()` para obtener el texto plano.

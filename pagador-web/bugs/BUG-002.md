# BUG-002: Convenio externo — ID mismatch en tabla migrada rompe selección y envío de documentos

## Fecha
2026-02-23

## Componentes involucrados
- `src/presentation/agreement/components/DocumentsStep/DocumentsTable/useHeadersRowsTable.tsx`
- `src/presentation/agreement/components/DocumentsStep/DocumentsTable/DocumentsTable.tsx`
- `src/presentation/agreement/hooks/useStepperReferences.ts`
- `src/presentation/agreement/utils/getFilteredDocuments.ts`

## Descripción del problema

Cuando el convenio seleccionado tiene `external_api_id` con valor externo (es externo, vía `isAllowedIdsMap`), el flujo de pago presenta dos síntomas:

1. **Selecciona automáticamente todo** — el sistema auto-selecciona un documento pero visualmente no se ve seleccionado en el grid (row aparece sin checkmark). Sin embargo, `isDisabledNextButton()` devuelve `false` porque `selectedDocument` tiene datos. El usuario puede estar confundido sobre qué documento está seleccionado.

2. **Al avanzar al paso 2, los IDs seleccionados no se envían** — al presionar "Siguiente", el request `GetShadowDocumentsRequest` llega con `instruction_ids_list: []` y `receivable_ids_list: []`, sin los documentos seleccionados.

## Causa raíz

Existe una inconsistencia entre cómo se construye el **ID de row** en `useHeadersRowsTable` vs cómo se construye el **ID en `filteredDocuments`** (vía `getFilteredDocuments`).

### `getFilteredDocuments.ts` (correcto)
```ts
const isExternalReceivableId = p.uuid ?? p.id;
id: isExternalReceivableId + "/" + DocumentReceivableTypeEnum.RECEIVABLE,
```
→ Para convenios externos: `"{uuid}/RECEIVABLE"` (UUID del documento externo)
→ Para convenios no externos: `"{numericId}/RECEIVABLE"` (id numérico)

### `useHeadersRowsTable.tsx` (incorrecto — path `!isSecondStep`)
```tsx
id: `${rec.id}/${rec.type}`,
```
→ Para **todos** los convenios: `"{numericId}/RECEIVABLE"` (siempre numérico, ignora `uuid`)

### Consecuencias del mismatch (cuando `MIGRATE_PAYMENT_DOCUMENT_TABLE = true`)

#### Síntoma 1 — Auto-select visual roto
El `useEffect` en `DocumentsTable` detecta `filteredDocuments.length === 1` y auto-selecciona:
```tsx
dispatch(setSelectedDocument([filteredDocuments[0].id]));  // → "{uuid}/RECEIVABLE"
```
Pero el grid tiene rows con ID `"{numericId}/RECEIVABLE"`. El `rowSelectionModel` apunta a un ID que no existe en las rows → **el grid no muestra ninguna row seleccionada visualmente**, aunque `selectedDocument` tiene datos.

#### Síntoma 2 — `useEffect` limpia selección válida
Cuando `FilterValues` monta y dispara `setFilteredDocuments`, `filteredDocuments` cambia → el `useEffect` de `DocumentsTable` se ejecuta:
```tsx
const tempSelected = filteredDocuments.find(
  (p) => p.id === selectedDocument[0],  // busca "{numericId}" en lista con IDs "{uuid}"
);
if (!tempSelected || !isRowSelectable(...)) {
  dispatch(setSelectedDocument(undefined));  // ← limpia selección válida
}
```
Si el usuario había seleccionado una row con ID numérico, `tempSelected` es `undefined` → se borra la selección.

#### Síntoma 3 — `buildShadowDocumentsToLink` no encuentra documentos
En `useStepperReferences.ts`:
```tsx
instruction_ids_list: buildShadowDocumentsToLink(
  documents.receivable_instruction_data_list,
  selectedDocument as string[],  // → ["numericId/RECEIVABLE"] (en path migrado)
),
```
Dentro de `buildShadowDocumentsToLink`:
```tsx
const sanitizedIds = selectedIds.map((p) => p.split("/")[0]);
// sanitizedIds = ["numericId"]  ← numérico

documentsToFilter.forEach((p) => {
  if (p.uuid && sanitizedIds.includes(p.uuid)) {  // "uuid_real" !== "numericId" → NO MATCH
    shadowDocumentsIds.push(p.uuid);
  }
});
// Retorna [] → request se envía sin documentos
```

## Comportamiento esperado
- El ID de cada row en el grid (migrado) debe coincidir con el ID en `filteredDocuments`
- Para convenios externos: ambos deben usar `(rec.uuid ?? rec.id)` como ID base
- `buildShadowDocumentsToLink` debe recibir UUIDs válidos y retornar los documentos seleccionados

## Fix sugerido

### `useHeadersRowsTable.tsx` — path `!isSecondStep`

Cambiar la construcción del ID de row para usar `uuid` cuando esté disponible (mirrors `getFilteredDocuments`):

```tsx
// Antes (incorrecto):
id: `${rec.id}/${rec.type}`,

// Después (correcto):
id: `${rec.uuid ?? rec.id}/${rec.type}`,
```

Ambos tipos (`ReceivableDataList` y `ReceivableInstructionDataList`) declaran `uuid?: string`, por lo que `rec.uuid` es accesible directamente sin casting.

### Impacto del fix
- Los IDs de rows en el grid migrado coincidirán con los IDs en `filteredDocuments`
- El auto-select del `useEffect` usará el mismo ID que el grid tiene → selección visual correcta
- El `useEffect` de validación encontrará el ID en `filteredDocuments` → no limpiará selección válida
- `buildShadowDocumentsToLink` recibirá el UUID correcto → `sanitizedIds` matcheará `p.uuid`

## Precondición para reproducir
- Feature flag `MIGRATE_PAYMENT_DOCUMENT_TABLE = true`
- Convenio con `external_api_id` que mapee a `true` en `isAllowedIdsMap` (valor `1`)
- Al menos un documento devuelto por el API externo con campo `uuid` presente

## Notas
- El path no migrado (`MIGRATE_PAYMENT_DOCUMENT_TABLE = false`) usa `finalFilteredDocuments` como rows → IDs ya son UUID-based via `getFilteredDocuments` → **no tiene este bug**
- El path `isSecondStep` de `useHeadersRowsTable` no tiene este problema (usa `rec.pay_ref_id` que es el ID correcto del PaymentDetail)
- El bug no existe en convenios no externos porque `p.uuid` es `undefined` → `getFilteredDocuments` usa `p.id` numérico igual que `useHeadersRowsTable`

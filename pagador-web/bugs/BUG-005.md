# 🐛 Bugfix Spec: full_name del pagador muestra pipe `|` en voucher para COMPRA_AHORA_PAGA_DESPUÉS

> **ID:** BUG-005
> **Fecha:** 2026-02-27
> **Proyecto:** pagador-web
> **Prioridad:** medium
> **Estado:** ready

---

## 1. Resumen

En `VoucherSumary`, cuando `payment_method_type === PWOMPI.COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN`, el campo `payerDetail.full_name` llega del API con un pipe `|` como separador interno, que se muestra literalmente en pantalla en lugar de un espacio.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---|---|
| `src/presentation/paymentLink/pages/Voucher/Voucher.tsx` | Construye `mappingDataSumary()` que pasa `payer_detail` a `VoucherSumary` — aquí se hace el fix |
| `src/presentation/paymentLink/components/Voucher/VoucherSumary/VoucherSumary.tsx` | Renderiza `{payerDetail.full_name}` en línea 160 |

### Dependencias clave

- Componente padre: `Voucher.tsx` (página)
- Store/slice: `paymentLink` — `voucher.payer_detail.full_name`, `transactionInfo.payment_method_type`
- Hooks: `useTypedSelector` vía `stateSelectorPayMethodContext`

## 3. Reproducción

### Pasos

1. Completar un pago con método `COMPRA_AHORA_PAGA_DESPUÉS` (`CODE_ORIGIN: "COMPRA_AHORA_PAGA_DESPUÉS"`)

## 4. Análisis de causa raíz

### Hipótesis

- [x] El API de COMPRA_AHORA_PAGA_DESPUÉS retorna `full_name` con `|` como separador de partes del nombre (ej. nombre `|` apellido)
- [ ] El pipe viene de otra fuente (mapeo previo, store, otro campo)

### Causa confirmada

El campo `voucher.payer_detail.full_name` llega del backend con `|` como separador (ej. `"Joaquin|Andres"`). El frontend lo renderiza sin sanitizar.
El API retorna `payment_method_type` con el valor `"BANCOLOMBIA_BNPL"` que corresponde a `PATCH` en el enum `PAYMENT_METHOD.WOMPI.COMPRA_AHORA_PAGA_DESPUÉS`, **no** `CODE_ORIGIN` ni `CODE_BACK`. Confirmado con console.log en producción.

**Archivo raíz:** `src/presentation/paymentLink/pages/Voucher/Voucher.tsx`
**Línea(s):** `205` — `payerDetail: payer_detail` dentro de `mappingDataSumary()`

---

## 5. Plan de fix

### Estrategia

Sanitizar `full_name` dentro de `mappingDataSumary()` en `Voucher.tsx`, condicionado al `payment_method_type`. En ese punto ya se tiene acceso tanto al `payment_method_type` (desde `transactionInfo`) como al `payer_detail`. No tocar `VoucherSumary` ni la interfaz `VoucherSumaryProps`.

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---|---|---|
| 1 | `Voucher.tsx` — función `mappingDataSumary()` | Cuando `payment_method_type === PWOMPI.COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN`, reemplazar todos los `|` de `payer_detail.full_name` por un espacio y hacer `.trim()` antes de asignarlo | SRP |

### Restricciones

- ❌ No modificar `VoucherSumary.tsx` (la prop `payerDetail` llega ya corregida)
- ❌ No modificar la interfaz `VoucherSumaryProps` ni `PayerDetail`
- ❌ No alterar el shape del store ni el thunk `getVoucher`
- ❌ No aplicar la sanitización a otros métodos de pago
- ✅ La sanitización va exclusivamente dentro de `mappingDataSumary()` en `Voucher.tsx`

### Cambios realizados

```diff
// Voucher.tsx — mappingDataSumary()
  const mappingDataSumary = () => {
    const { transaction_detail, payer_detail, agreement_information } = voucher;
    const { total_amount, reference, currency } = transaction_detail;
    const total_amount_in_cents =
      total_amount + (transaction_detail?.commission_amount_in_cents || 0);
+   const sanitizedFullName =
+     payment_method_type === PWOMPI.COMPRA_AHORA_PAGA_DESPUÉS.CODE_BACK
+       ? payer_detail.full_name.replace(/\|/g, " ").trim()
+       : payer_detail.full_name;
    const sumary: VoucherSumaryProps = {
      id: reference,
      amount: amountInCop(total_amount_in_cents.toString()),
      currency: currency,
      agreement: agreement_information?.name,
      references: paymentDetail.references.filter(
        (p) => p.show_to_payer_detail && !p.is_entry_reference,
      ),
-     payerDetail: payer_detail,
+     payerDetail: { ...payer_detail, full_name: sanitizedFullName },
      handleDownloadClick: handleDownloadClick,
    };
    return sumary;
  };
```

### Lecciones aprendidas

- **Patrón del bug:** API retorna datos con separadores no estándar (`|`) para un método de pago específico; el frontend debe sanitizar en el punto de mapeo, no en el componente de presentación.
- **Skill relacionada:** `components/VoucherSumary.md`, `components/Voucher.md`
- **Regla nueva:** La sanitización de datos del API va en el adapter/mapping (`mappingDataSumary`, `mappingDataTransaction`, etc.), nunca en el componente leaf.

# 🐛 Bugfix Spec: Disclaimer oculto en FormPersonalData para DAVIPLATA y SUMAS_PAY

> **ID:** BUG-004
> **Fecha:** 2026-02-24
> **Proyecto:** pagador-web
> **Prioridad:** medium
> **Estado:** draft

---

## 1. Resumen

El bloque disclaimer en `FormPersonalData` no se muestra cuando `transactionCourse.business_detail.id` está incluido en los arrays `paymentMethodsNameById.DAVIPLATA` o `paymentMethodsNameById.SUMAS_PAY`, debido a lógica invertida y problema de precedencia de operadores en el JSX.

## 3. Reproducción

### Pasos

1. Abrir un payment link cuyo `transactionCourse.business_detail.id` esté incluido en `paymentMethodsNameById.DAVIPLATA` o `paymentMethodsNameById.SUMAS_PAY`
2. Observar el formulario de datos personales (`FormPersonalData`)
3. El bloque disclaimer (ícono `InfoIcon` + texto `disclaimerText`) no aparece

### Comportamiento actual

El disclaimer nunca se muestra cuando `business_detail.id` pertenece a DAVIPLATA o SUMAS_PAY.

Causa técnica — dos problemas combinados en `FormPersonalData.tsx` líneas 205-222:

```tsx
// PROBLEMA 1: isNotAPaymentMethod devuelve true cuando NO está en el array
//             → lógica invertida respecto a lo que se quiere
// PROBLEMA 2: precedencia incorrecta — el || suelto cortocircuita con `true`
//             sin llegar al JSX
{isNotAPaymentMethod(
  transactionCourse.business_detail.id,
  paymentMethodsNameById.SUMAS_PAY ?? [],
) ||
  (isNotAPaymentMethod(
    transactionCourse.business_detail.id,
    paymentMethodsNameById.DAVIPLATA ?? [],
  ) && (
    <div className="form-personal-data__disclaimer">...</div>
  ))}
```

### Comportamiento esperado

El disclaimer debe mostrarse cuando `transactionCourse.business_detail.id` está incluido (`.includes()`) en `paymentMethodsNameById.DAVIPLATA` **o** en `paymentMethodsNameById.SUMAS_PAY`.

---

## 4. Análisis de causa raíz

### Hipótesis

- [x] Uso incorrecto de `isNotAPaymentMethod` (negación) en lugar de `isPaymentMethod` (afirmación)
- [x] Precedencia de operadores incorrecta: `A || (B && JSX)` en lugar de `(A || B) && JSX`

### Causa confirmada

```
isNotAPaymentMethod devuelve !array.includes(id).
Si id está en SUMAS_PAY → isNotAPaymentMethod devuelve false
Si id NO está en SUMAS_PAY → devuelve true, pero truthy cortocircuita el ||
y nunca llega al JSX.
En ningún caso el disclaimer se renderiza correctamente.
```

**Archivo raíz:** `src/presentation/paymentLink/components/Main/FormPayment/FormPersonalData/FormPersonalData.tsx`
**Línea(s):** `205-222`

---

## 5. Plan de fix

### Estrategia

Reemplazar las llamadas a `isNotAPaymentMethod` por `isPaymentMethod` y corregir la precedencia agrupando ambas condiciones antes del `&&` JSX.

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `FormPersonalData.tsx` líneas 205-222 | Sustituir `isNotAPaymentMethod` por `isPaymentMethod` y envolver ambas condiciones en paréntesis antes del `&& <div>` | SRP |

### Restricciones

- ❌ No modificar `isNotAPaymentMethod` ni `isPaymentMethod` (usados en otros lugares)
- ❌ No alterar el shape del store ni otros renders del componente
- ❌ No cambiar archivos fuera del scope listado
- ✅ Se puede importar `isPaymentMethod` desde el mismo módulo ya existente

### Lecciones aprendidas

- **Patrón del bug:** Uso de función negada (`isNotAPaymentMethod`) donde se necesita la afirmada, combinado con precedencia de operadores incorrecta en JSX condicional
- **Skill relacionada:** `components/FormPersonalData.md`
- **Regla nueva:** Cuando el disclaimer debe mostrarse para métodos específicos, usar `isPaymentMethod`; cuando debe ocultarse para métodos específicos, usar `isNotAPaymentMethod`

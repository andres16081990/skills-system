# 🐛 Bugfix Spec: Label incorrecto para `external_identifier` según método de pago

> **ID:** BUG-003
> **Fecha:** 2026-02-24
> **Proyecto:** pagador-web
> **Prioridad:** medium
> **Estado:** resolved

---

## 1. Resumen

En `Voucher.tsx`, cuando `external_identifier` está presente, el label que se muestra en la tabla de transacción siempre usa la clave `nequiId` ("ID Nequi"), sin importar el método de pago. Para DAVIPLATA, SUMAS_PAY y COMPRA_AHORA_PAGA_DESPUÉS debería mostrarse el label `tiketId` ("Ticket ID").

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/paymentLink/pages/Voucher/Voucher.tsx` | Componente afectado — función `mappingDataTransaction()` |
| `src/presentation/paymentLink/components/Main/FormPayment/types/paymentMethods.ts` | Constantes `PAYMENT_METHOD` con los `CODE_ORIGIN` de cada método |
| `src/i18n/languages/es.json` | Archivo de traducciones — sección `properties.payment_link.pages.voucher` |

### Dependencias clave

- Store/slice: `PaymentLinkReducer` — `transactionInfo.payment_method_type`
- Constantes: `PAYMENT_METHOD.DAVIPLATA.CODE_ORIGIN`, `PAYMENT_METHOD.SUMAS_PAY.CODE_ORIGIN`, `PAYMENT_METHOD.COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN`, `PAYMENT_METHOD.NEQUI.CODE_ORIGIN`
- Traducción actual afectada: `nequiId` → `"ID Nequi"`

---

## 3. Reproducción

### Pasos

1. Completar un pago con método DAVIPLATA, SUMAS_PAY o COMPRA_AHORA_PAGA_DESPUÉS.
2. Navegar a la página de Voucher con una transacción que tenga `external_identifier` en los datos del voucher.
3. Observar el label que aparece junto al valor del `external_identifier` en la tabla de transacción.

### Comportamiento actual

```
El label mostrado es siempre "ID Nequi" (clave nequiId),
independientemente del método de pago usado.
```

### Comportamiento esperado

```
- payment_method_type === NEQUI.CODE_ORIGIN            → label: "ID Nequi"    (clave nequiId)
- payment_method_type === DAVIPLATA.CODE_ORIGIN         → label: "Ticket ID"   (clave tiketId)
- payment_method_type === SUMAS_PAY.CODE_ORIGIN         → label: "Ticket ID"   (clave tiketId)
- payment_method_type === COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN → label: "Ticket ID" (clave tiketId)
```
En todos los casos el label y el valor deben ser visibles cuando `external_identifier` existe.

---

## 4. Análisis de causa raíz

### Hipótesis

- [x] `mappingDataTransaction()` usa siempre `translate("nequiId")` sin ramificar por `payment_method_type`

### Causa confirmada

```
En Voucher.tsx línea ~244-248, el bloque que agrega external_identifier al array
usa incondicionalmente translate("nequiId"). No existe validación del payment_method_type
para elegir entre nequiId y tiketId.
```

**Archivo raíz:** `src/presentation/paymentLink/pages/Voucher/Voucher.tsx`
**Línea(s):** ~244–248 (bloque `if (external_identifier)`)

---

## 5. Plan de fix

### Estrategia

Dentro de `mappingDataTransaction()`, al construir el item de `external_identifier`, determinar la clave de traducción según `payment_method_type`: usar `"tiketId"` para DAVIPLATA, SUMAS_PAY y COMPRA_AHORA_PAGA_DESPUÉS; mantener `"nequiId"` para NEQUI. Agregar la clave `tiketId` en el archivo i18n.

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `src/i18n/languages/es.json` | Agregar clave `"tiketId": "Ticket ID"` dentro de `properties.payment_link.pages.voucher` | SRP |
| 2 | `src/presentation/paymentLink/pages/Voucher/Voucher.tsx` | En `mappingDataTransaction()`, reemplazar `translate("nequiId")` por una expresión condicional que evalúe `payment_method_type` contra `PAYMENT_METHOD.DAVIPLATA.CODE_ORIGIN`, `PAYMENT_METHOD.SUMAS_PAY.CODE_ORIGIN` y `PAYMENT_METHOD.COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN` → usa `"tiketId"`; en cualquier otro caso usa `"nequiId"` | OCP |

### Restricciones

- ❌ No modificar la lógica de ningún otro campo de `mappingDataTransaction()`
- ❌ No cambiar la clave `nequiId` ni su valor en i18n
- ❌ No tocar otros archivos fuera del scope listado
- ✅ Se permite agregar claves nuevas en i18n sin alterar las existentes

## 8. Resolución

### Fecha de resolución

2026-02-24

### Cambios realizados

```diff
// es.json
+ "tiketId": "Ticket ID",

// Voucher.tsx — mappingDataTransaction()
- label: `${translate("nequiId")}`,
+ label: `${translate(
+   payment_method_type === PWOMPI.DAVIPLATA.CODE_ORIGIN ||
+   payment_method_type === PWOMPI.SUMAS_PAY.CODE_ORIGIN ||
+   payment_method_type === PWOMPI.COMPRA_AHORA_PAGA_DESPUÉS.CODE_ORIGIN
+     ? "tiketId"
+     : "nequiId"
+ )}`,
```

### Lecciones aprendidas

- **Patrón del bug:** Label hardcodeado a un método de pago específico sin ramificar por `payment_method_type`
- **Skill relacionada:** `components/Voucher.md`
- **Regla nueva:** Al agregar campos condicionales en `mappingDataTransaction`, siempre evaluar `payment_method_type` para labels que son específicos de un método

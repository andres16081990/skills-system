# 🐛 Bugfix Spec: DateRangePicker del dashboard no restaura las cards al regresar de conciliación o documentos de cobro

> **ID:** BUG-004
> **Fecha:** 2026-02-26
> **Proyecto:** recaudador-web
> **Prioridad:** high
> **Estado:** resolved

---

## 1. Resumen

Cuando el usuario aplica un filtro por `DateRangePicker` en el dashboard, navega a conciliación o documentos de cobro abriendo una card, y regresa al home, las cards vuelven a mostrar datos con fechas vacías (`DashboardHomeInitial`) en lugar de mantener el rango de fechas aplicado.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/home/pages/hooks/useDashboardTab.ts` | Hook principal del dashboard — contiene la causa raíz |
| `src/presentation/home/pages/components/dashboardTabComponent/DashboardTabComponent.tsx` | Componente que usa `useDashboardTab` y renderiza el `DateRangePicker` |
| `src/presentation/receivable/hooks/useDashboardRedirect.tsx` | Setea `keepDashboardConfiguration(true)` cuando el usuario regresa al home |
| `src/presentation/conciliation/state/ConciliationReducer.ts` | Contiene `setKeepDashboardConfiguration`, `setConciliationDashboardConfiguration` |

### Dependencias clave

- Hook principal: `useDashboardTab`
- Store/slice: `ConciliationReducer` — campos `keepDashboardConfiguration`, `dashboardConfiguration`
- Hooks: `useDashboardRedirect` — activa el flag de restauración al regresar
- Estado local: `dashboardDate` (fecha "confirmada" que dispara la API) vs `dashboardDateRange` (fecha "pendiente" del picker antes de confirmar)

---

## 3. Reproducción

### Pasos

1. Ir al **Home / Dashboard**
2. Seleccionar un rango de fechas con el **DateRangePicker**
3. Hacer clic en el botón de confirmación ("Aplicar" / `handleOnSubmit`) → las cards se actualizan con el rango seleccionado
4. Hacer clic en una card para navegar a **Conciliación** o **Documentos de Cobro**
5. Hacer clic en el botón "Volver al dashboard"
6. Observar las cards en el home

### Comportamiento actual

Las cards se recargan con las fechas vacías (`initialDate: ""`, `finalDate: ""`), mostrando datos del período por defecto en lugar del rango de fechas que el usuario había seleccionado.

### Comportamiento esperado

Al regresar al home, las cards deben mostrar los mismos valores que tenían cuando el usuario aplicó el `DateRangePicker`, replicando el comportamiento que ya funciona correctamente con los filtros de **tabs** (`TabContainer`).

---

## 4. Análisis de causa raíz

### Hipótesis

- [x] `setValuesByConfiguration()` restaura `dashboardDateRange` (estado "pendiente" del picker) pero NO restaura `dashboardDate` (estado "confirmado" que dispara la API)
- [ ] `keepDashboardConfiguration` no llega en `true` al montar el home tras regresar con el botón de volver
- [ ] Race condition entre el cleanup del `useEffect` de mount y el dispatch de `useDashboardRedirect`

### Causa confirmada

El `useEffect` de mount en `useDashboardTab` (líneas 106-120) siempre llama `setDashboardDate(DashboardHomeInitial)` primero. Luego, si `keepDashboardConfiguration` es `true`, llama a `setValuesByConfiguration()`.

**Ruta de tabs (funciona ✅):**
```
setValuesByConfiguration()
  → setTabValue(savedTabValue)
  → [tab useEffect] setDashboardDate(getDatesByTabvalue(tabValue))
  → [API useEffect] dispatch(getDashboardHomeReport(datesCorrects))
```

**Ruta de DateRangePicker (rota ❌):**
```
setDashboardDate(DashboardHomeInitial)   ← se llama primero en el mount effect
setValuesByConfiguration()
  → setDashboardDateRange(savedRange)   ← solo actualiza el estado "pendiente" del picker
  → ❌ dashboardDate permanece en DashboardHomeInitial
  → [API useEffect] dispatch(getDashboardHomeReport({ initialDate: "", finalDate: "" }))
```

**Archivo raíz:** `src/presentation/home/pages/hooks/useDashboardTab.ts`
**Línea(s):** 122-132 (`setValuesByConfiguration`) y 106-120 (mount `useEffect`)

```typescript
// CAUSA: solo se actualiza dashboardDateRange, no dashboardDate
const setValuesByConfiguration = () => {
  if (dashboardConfiguration.tabValue !== null) {
    setTabValue(dashboardConfiguration.tabValue);
    setDashboardDateRange(DashboardHomeInitial);
  } else {
    // ❌ Falta llamar a setDashboardDate aquí
    setDashboardDateRange({
      finalDate: dashboardConfiguration.metaData.finalDate,
      initialDate: dashboardConfiguration.metaData.initialDate,
    });
  }
};
```

## 5. Plan de fix

### Estrategia

En `setValuesByConfiguration()`, en el bloque `else` (ruta DateRangePicker), también llamar a `setDashboardDate` con el rango guardado en `dashboardConfiguration.metaData`. Esto hará que el `useEffect` de la API se dispare con las fechas correctas, igual que ocurre con la ruta de tabs.

React 18 batching asegura que el `setDashboardDate(DashboardHomeInitial)` del mount effect y el `setDashboardDate(savedRange)` de `setValuesByConfiguration` se resuelvan en el mismo ciclo de render, ganando el último valor asignado (el rango guardado).

### Cambios requeridos

| # | Archivo | Cambio | Principio |
|---|---------|--------|-----------|
| 1 | `src/presentation/home/pages/hooks/useDashboardTab.ts` | En `setValuesByConfiguration()`, agregar `setDashboardDate({ finalDate: dashboardConfiguration.metaData.finalDate, initialDate: dashboardConfiguration.metaData.initialDate })` en el bloque `else` (ruta DateRangePicker) | SRP |

### Restricciones

- ❌ No modificar `DashboardTabComponent.tsx` — el fix es solo en el hook
- ❌ No modificar el flujo de tabs — el bloque `if (tabValue !== null)` no se toca
- ❌ No alterar `useDashboardRedirect`, `handleCardOnClick` ni `handleOnSubmit`
- ❌ No cambiar el shape del store ni el reducer
- ✅ Solo agregar `setDashboardDate` en el `else` de `setValuesByConfiguration`
- ✅ `useDashboardTab` es usado en un solo lugar (`DashboardTabComponent`), se puede modificar directamente según GLOBAL_RULES

### Fecha de resolución

2026-02-26

### Cambios realizados

**Archivo:** `src/presentation/home/pages/hooks/useDashboardTab.ts`

```diff
  const setValuesByConfiguration = () => {
    if (dashboardConfiguration.tabValue !== null) {
      setTabValue(dashboardConfiguration.tabValue);
      setDashboardDateRange(DashboardHomeInitial);
    } else {
+     setDashboardDate({
+       finalDate: dashboardConfiguration.metaData.finalDate,
+       initialDate: dashboardConfiguration.metaData.initialDate,
+     });
      setDashboardDateRange({
        finalDate: dashboardConfiguration.metaData.finalDate,
        initialDate: dashboardConfiguration.metaData.initialDate,
      });
    }
  };
```

### Lecciones aprendidas

- **Patrón del bug:** Asimetría entre la ruta de restauración de tabs y la de DateRangePicker. Los tabs usan `setTabValue` que indirectamente actualiza `dashboardDate` vía su propio `useEffect`. El DateRangePicker solo actualizaba `dashboardDateRange` (estado "pendiente"), dejando `dashboardDate` en el valor inicial y disparando la API con fechas vacías.
- **Regla nueva:** Cuando existan dos caminos de restauración de estado (tab vs. picker), verificar que ambos actualicen el estado que dispara efectos secundarios críticos (en este caso `dashboardDate` → API call). El estado "pendiente" de un picker no es suficiente; también hay que confirmar el estado "comprometido".

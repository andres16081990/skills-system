# 🐛 Bugfix Spec: "Restablecer filtros" del dashboard no resetea las cards ni el estado de Redux

> **ID:** BUG-006
> **Fecha:** 2026-02-26
> **Proyecto:** recaudador-web
> **Prioridad:** high
> **Estado:** resolved

---

## 1. Resumen

Al hacer clic en "Restablecer filtros" (`handleResetFilters`), el dashboard no se comporta igual que al cargar por primera vez: en el caso de que el usuario tenga un **DateRangePicker** aplicado, las cards no se recargan con el período por defecto, y en ambos casos (tab o DateRangePicker) el estado de Redux (`dashboardConfiguration`) queda con los valores del filtro anterior.

---

## 2. Contexto

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/presentation/home/pages/hooks/useDashboardTab.ts` | Hook afectado — contiene `handleResetFilters` y el `useEffect` de primer mount |
| `src/presentation/home/pages/components/dashboardTabComponent/DashboardTabComponent.tsx` | Componente que dispara `handleResetFilters` al hacer clic en el `LinkText` |
| `src/presentation/home/pages/models/Home.interface.ts` | Define `initialDashboardConfiguration` — el estado inicial de Redux |

### Dependencias clave

- Hook: `useDashboardTab`
- Estado local: `dashboardDate` (dispara la API), `dashboardDateRange` (estado pendiente del picker), `tabValue`, `resetCalendar`
- Estado Redux: `dashboardConfiguration` (`tabValue` + `metaData`)
- Acción Redux: `setConciliationDashboardConfiguration`

---

## 3. Reproducción

### Caso A — DateRangePicker aplicado

1. Ir al **Home / Dashboard**
2. Seleccionar un rango de fechas con el **DateRangePicker** y confirmar
3. Verificar que las cards muestran datos filtrados
4. Hacer clic en **"Restablecer filtros"**

**Comportamiento actual:** Las cards NO se recargan. Siguen mostrando los datos del rango de fechas aplicado.

**Por qué:** `setTabValue(null)` es un no-op porque `tabValue` ya era `null`. El `useEffect` del tab no se dispara, `dashboardDate` nunca se resetea a `DashboardHomeInitial`, y el `useEffect` de la API no se ejecuta.

---

### Caso B — Tab seleccionado

1. Ir al **Home / Dashboard**
2. Seleccionar un tab (ej. "Esta semana")
3. Verificar que las cards se actualizan
4. Hacer clic en **"Restablecer filtros"**

**Comportamiento actual:** Las cards SÍ se recargan (porque `setTabValue(null)` cambia el estado y dispara el `useEffect` del tab → `setDashboardDate(DashboardHomeInitial)` → API). Pero `dashboardConfiguration` en Redux mantiene el `tabValue` anterior, lo que puede causar comportamientos incorrectos si el usuario navega a otra pantalla y regresa.

---

### Comportamiento esperado (ambos casos)

Igual al primer mount sin `keepDashboardConfiguration`:
1. Las cards se recargan con el período por defecto (`DashboardHomeInitial`)
2. El picker se limpia visualmente
3. Ningún tab queda seleccionado
4. `dashboardConfiguration` en Redux queda en `initialDashboardConfiguration`

---

## 4. Análisis de causa raíz

### Comparación: primer mount vs. `handleResetFilters`

| Acción | Primer mount | `handleResetFilters` actual |
|--------|-------------|--------------------------|
| `setDashboardDate(DashboardHomeInitial)` | ✅ siempre | ❌ nunca (solo indirectamente si tabValue cambia) |
| `setDashboardDateRange(DashboardHomeInitial)` | ✅ (implícito — estado inicial) | ✅ |
| `setResetCalendar(true)` | — | ✅ |
| `setTabValue(null)` | — | ✅ |
| `dispatch(setConciliationDashboardConfiguration(initialDashboardConfiguration))` | ✅ | ❌ nunca |

### Causa confirmada

`handleResetFilters` no llama a `setDashboardDate(DashboardHomeInitial)` directamente, sino que delega en el `useEffect` del tab vía `setTabValue(null)`. Cuando `tabValue` ya es `null` (caso DateRangePicker), el `useEffect` no se dispara y `dashboardDate` no se resetea → la API no se vuelve a llamar con fechas vacías.

Adicionalmente, `dispatch(setConciliationDashboardConfiguration({ ...initialDashboardConfiguration }))` nunca se llama en el reset, dejando el estado Redux desincronizado.

**Archivo raíz:** `src/presentation/home/pages/hooks/useDashboardTab.ts`
**Línea(s):** 100-104

```typescript
// ACTUAL — incompleto
const handleResetFilters = () => {
  setDashboardDateRange(DashboardHomeInitial);  // ✅
  setResetCalendar(true);                       // ✅
  setTabValue(null);                            // ✅ solo funciona si tabValue !== null
  // ❌ falta setDashboardDate(DashboardHomeInitial)
  // ❌ falta dispatch(setConciliationDashboardConfiguration(initialDashboardConfiguration))
};
```

## 8. Resolución

<!-- Completar DESPUÉS del fix -->

### Fecha de resolución

2026-02-26

### Cambios realizados

**Archivo:** `src/presentation/home/pages/hooks/useDashboardTab.ts`

```diff
  const handleResetFilters = () => {
+   setDashboardDate(DashboardHomeInitial);
    setDashboardDateRange(DashboardHomeInitial);
    setResetCalendar(true);
    setTabValue(null);
+   dispatch(
+     setConciliationDashboardConfiguration({
+       ...initialDashboardConfiguration,
+     }),
+   );
  };
```

### Lecciones aprendidas

- **Patrón del bug:** Función de reset que delega en efectos secundarios encadenados (`setTabValue(null)` → `useEffect` del tab → `setDashboardDate`) en lugar de ser explícita. Cuando la precondición del efecto no se cumple (tabValue ya era null), el reset queda incompleto.
- **Regla nueva:** Una función de reset debe reproducir explícitamente el estado inicial completo — tanto local como Redux — sin depender de efectos encadenados que pueden no dispararse.

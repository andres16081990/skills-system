# 🐛 Bugfix Spec: Filtro "por vencerse" desde dashboard rompe paginación en PaymentDocumentMain

> **ID:** BUG-003
> **Fecha:** 2026-02-25
> **Proyecto:** recaudador-web
> **Prioridad:** critical
> **Estado:** resolved

---

## 1. Resumen

Al entrar a `PaymentDocumentMain` desde el dashboard con el filtro "por vencerse" activo, la paginación no funciona correctamente porque en algún punto del flujo se está seteando a `false` una referencia que la paginación necesita activa.

### Dependencias clave

- El bug **no ocurre** cuando se accede a `PaymentDocumentMain` directamente (sin venir del dashboard)

---

## 3. Reproducción

### Pasos

1. Ir al **Dashboard**
2. Hacer clic en el card/acceso del filtro **"por vencerse"**
3. Esperar a que cargue `PaymentDocumentMain` con ese filtro aplicado
4. Intentar **paginar** los resultados

### Comportamiento actual

La paginación no funciona correctamente. Una referencia interna está siendo seteada a `false` durante el flujo de entrada desde el dashboard, lo que impide que la paginación opere como debería.

### Comportamiento esperado

La paginación debe funcionar correctamente. La referencia problemática **no debe resetearse a `false`** durante el flujo de entrada desde el dashboard con el filtro "por vencerse".

## 5. Plan de fix

### Estrategia

Agregar una guarda que evalúe **ambas condiciones simultáneamente** antes de ejecutar el reset de la referencia:
1. ¿Viene del dashboard? (`fromDashboardCardSelected === true`)
2. ¿El filtro activo es "por vencerse"?

Solo si **las dos condiciones son verdaderas** se debe omitir el seteo a `false` de la referencia. Si el usuario interactúa con los filtros manualmente, el reset sí puede ejecutarse normalmente.

### Cambios realizados

**Archivo:** `src/presentation/paymentDocument/hooks/usePaymentDocumentReport.ts`

```diff
  dispatch(setChangedFilters(filtersChanged));
  dispatch(setIdleState());
- changeFiltersRef.current = false;
+ const isFromDashboardWithSoonDue =
+   state?.source === ComponentSource.DASHBOARD &&
+   state?.cardSelected === CardSelected.PENDING_COLLECTION_IN_CENTS;
+ if (!isFromDashboardWithSoonDue) {
+   changeFiltersRef.current = false;
+ }
```

### Lecciones aprendidas

- **Patrón del bug:** Reset incondicional de `useRef` en un efecto que se dispara tras cada éxito de API, incluyendo paginación. La referencia `changeFiltersRef.current` se usaba como señal de contexto (dashboard + filtro especial) en `buildFilterToSearch`, pero era destruida después del primer load exitoso.
- **Skill relacionada:** `components/PaymentDocumentMain/PaymentDocumentMain.md`
- **Regla nueva:** Cuando una `ref` actúa como flag de contexto de navegación (ej: `fromDashboard`), no resetearla en efectos de éxito de API que se disparan durante la paginación. Solo resetearla en interacciones explícitas del usuario (filtros manuales, limpieza).

### ¿Se debe actualizar alguna skill?

- [x] Sí → Skill: `PaymentDocumentMain` | Cambio: Documentar en "Puntos frágiles" el patrón de `changeFiltersRef` como flag de contexto de navegación y la regla de no resetearlo en paginación cuando viene de dashboard + "por vencerse".

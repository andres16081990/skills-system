# 🐛 Bugfix Spec: Botón "Generar código de pago" no se deshabilita para acuerdos con external_api_id === 1

> **ID:** BUG-008
> **Fecha:** 2026-02-27
> **Proyecto:** recaudador-web
> **Prioridad:** high
> **Estado:** resolved

---

## 1. Resumen

En la tabla de `ReceivableTab`, el botón de generar enlace de pago (`GeneratePaymentCodeOption`) permanece habilitado incluso cuando el acuerdo activo tiene `external_api_id === 1`. Debe estar deshabilitado en ese caso.

---

## 2. Contexto

### Cadena de componentes / hooks involucrados

```
ReceivableTab
  └─ useDataDocumentTable (hook)
       └─ getMoreActions (función)
            └─ GeneratePaymentCodeOption (componente)
                 └─ useGeneratePaymentCode (hook)
                      └─ getIsDisabled() / isDisabled
```

### Archivos involucrados

| Archivo | Rol |
|---------|-----|
| `src/domain/model/response/AgreementDetailResponse.ts` | Interfaz `Agreement` del detalle — se agregó `external_api_id?: number` (campo que no devuelve el endpoint de detalle, pero mantiene consistencia de tipo) |
| `src/presentation/receivable/hooks/useDataDocumentTable.tsx` | Lee `agreements` del store, calcula `isExternalApiDisabled` y lo pasa en `moreActionsProps` |
| `src/presentation/receivable/hooks/useMapDataDocumentTable.tsx` | Lee `agreements` del store, calcula `isExternalApiDisabled` y lo pasa en la llamada a `getMoreActions` del DataGrid |
| `src/presentation/receivable/components/GetMoreActions/GetMoreActions.tsx` | Recibe `isExternalApiDisabled` como prop y lo pasa a `GeneratePaymentCodeOption` |
| `src/presentation/receivable/types/receivable.d.ts` | Define `GetMoreActionsProps` — se agregó `isExternalApiDisabled?: boolean` |
| `src/presentation/receivable/components/GeneratePaymentCodeOption/GeneratePaymentCodeOption.tsx` | Componente del botón — `getIsDisabled()` evalúa `isExternalApiDisabled` primero |

### Dependencias clave

- Store/slice: `AgreementState` → `agreementDetail.agreement` (tipo `AgreementDetailResponse.Agreement`) y `agreements` (tipo `AgreementSimpleResponse.Agreement[]`)
- Hook consumidor: `useGeneratePaymentCode` — no se modifica
- Modelo: `AgreementDetailResponse.Agreement` no tiene `external_api_id` en la respuesta del endpoint de detalle — solo lo devuelve el endpoint de lista, disponible en `AgreementState.agreements`

---

## 3. Reproducción

### Pasos

1. Navegar al detalle de un acuerdo cuyo `external_api_id === 1`
2. Ir a la pestaña de Receivables o Instructions en `ReceivableTab`
3. Observar el botón de enlace de pago (ícono de link) en cualquier fila de la tabla

### Comportamiento actual

El botón `GeneratePaymentCodeOption` está habilitado y permite generar el código de pago.

### Comportamiento esperado

El botón debe aparecer **deshabilitado** (igual que cuando `isDisabled = true`) cuando el acuerdo activo tiene `external_api_id === 1`.

---

## 4. Análisis de causa raíz

### Causa confirmada

**Problema 1 — fuente de datos incorrecta**: el endpoint de detalle del acuerdo (`getAgreementDetailByCode`) **no devuelve `external_api_id`** en su respuesta. Por eso `agreementDetail.agreement.external_api_id` siempre es `undefined`, haciendo que `isExternalApiDisabled` nunca sea `true`. El campo `external_api_id` solo está disponible en el endpoint de lista de acuerdos, mapeado en `AgreementState.agreements` (tipo `AgreementSimpleResponse.Agreement`).

Esto se confirmó observando las props del componente en el inspector:
```
isExternalApiDisabled: false   ← aunque el acuerdo tiene external_api_id === 1
```

**Problema 2 — dato no propagado**: `getMoreActions` y `GeneratePaymentCodeOption` no tienen ningún mecanismo para recibir este dato.

**Archivo raíz:** `useDataDocumentTable.tsx` y `useMapDataDocumentTable.tsx` — leen `agreementDetail.agreement` para construir el `agreement` que pasan a `getMoreActions`, pero ese objeto nunca tiene `external_api_id` porque el endpoint de detalle no lo incluye.

---

## 5. Plan de fix

### Estrategia

La fuente correcta de `external_api_id` es `AgreementState.agreements` (lista de acuerdos, endpoint de lista). En ambos hooks que construyen `moreActionsProps` se lee `agreements` del store, se busca el acuerdo activo por `agreement_code`, y se calcula `isExternalApiDisabled` localmente. Ese booleano se propaga como nuevo prop por la cadena hasta `GeneratePaymentCodeOption`, donde se evalúa al inicio de `getIsDisabled()`.

No se toca `useGeneratePaymentCode` porque el cambio es específico al contexto de la tabla (vía `GeneratePaymentCodeOption`).

### Cambios requeridos (real)

| # | Archivo | Cambio |
|---|---------|--------|
| 1 | `src/domain/model/response/AgreementDetailResponse.ts` | Agregar `external_api_id?: number` a la interfaz (consistencia de tipo, no lo devuelve el endpoint) |
| 2 | `src/presentation/receivable/types/receivable.d.ts` | Agregar `isExternalApiDisabled?: boolean` a `GetMoreActionsProps` |
| 3 | `src/presentation/receivable/hooks/useDataDocumentTable.tsx` | Leer `agreements` del store, calcular `isExternalApiDisabled` y pasarlo en los dos `moreActionsProps` (instructions y receivables) |
| 4 | `src/presentation/receivable/hooks/useMapDataDocumentTable.tsx` | Leer `agreements` del store, calcular `isExternalApiDisabled` y pasarlo en el `getMoreActions` del `renderCell` del DataGrid |
| 5 | `src/presentation/receivable/components/GetMoreActions/GetMoreActions.tsx` | Recibir `isExternalApiDisabled` del destructuring y pasarlo a `GeneratePaymentCodeOption` (eliminar cálculo interno basado en `agreement.external_api_id`) |
| 6 | `src/presentation/receivable/components/GeneratePaymentCodeOption/GeneratePaymentCodeOption.tsx` | Agregar prop `isExternalApiDisabled?: boolean` y guard al inicio de `getIsDisabled()` |

### Código del fix real

**`useDataDocumentTable.tsx` y `useMapDataDocumentTable.tsx`** (mismo patrón en ambos):
```ts
const { agreementDetail, agreements } = useTypedSelector<AgreementState>(
  stateSelectorAgreement,
);
const agreement = agreementDetail.agreement;
const isExternalApiDisabled =
  agreements.find((a) => a.agreement_code === agreement.code)
    ?.external_api_id === 1;
```

**`GetMoreActions.tsx`** — recibe el valor calculado, ya no lo calcula internamente:
```ts
export const getMoreActions = ({
  // ...
  isExternalApiDisabled,
}: GetMoreActionsProps) => {
  // ← sin cálculo local
  return (
    <GeneratePaymentCodeOption
      isExternalApiDisabled={isExternalApiDisabled}
      // ...
    />
  );
};
```

**`GeneratePaymentCodeOption.tsx`**:
```ts
const getIsDisabled = () => {
  if (isExternalApiDisabled) return true;   // ← guard primero
  const isShadowDisabled = disableShadowDocuments();
  // ... resto sin cambios
};
```

### Restricciones

- ❌ No modificar `useGeneratePaymentCode`
- ❌ No modificar `GeneratePaymentCodeLink` ni `ReceivableForm`
- ❌ No calcular `isExternalApiDisabled` desde `agreementDetail.agreement` — el endpoint de detalle no devuelve `external_api_id`
- ✅ Siempre leer `external_api_id` desde `AgreementState.agreements` buscando por `agreement_code`

### Lecciones aprendidas

- **Patrón del bug:** El endpoint de detalle de acuerdo no devuelve todos los campos del acuerdo — `external_api_id` solo está disponible en el endpoint de lista. Intentar leerlo desde `agreementDetail.agreement` siempre da `undefined`.
- **Skill relacionada:** `components/useGeneratePaymentCode/useGeneratePaymentCode.md`
- **Regla nueva:** Para leer `external_api_id` de un acuerdo activo, buscar en `AgreementState.agreements` por `agreement_code`. No confiar en `agreementDetail.agreement` para este campo.
